Все методы реализованы через следующую связь: клиент -> роутер -> сервис -> роутер -> клиент.

1. GET /weather/current — Получение текущей погоды

Параметры запроса:
- `latitude` (float, обязательный) — широта, от -90.0 до 90.0
- `longitude` (float, обязательный) — долгота, от -180.0 до 180.0

Логика работы:
    1. Роутер `get_current_weather` принимает координаты
    2. Вызывается сервис `get_current_weather()` из `services/weather_api.py`
    3. Сервис отправляет запрос к Open-Meteo API
    4. Ответ форматируется и возвращается клиенту


2. GET insert_city_ib_db - добавление города в БД для отслеживания погоды и автоматического обновления данных каждые 15 минут.

Параметры запроса:
- `city` (str, обязательный) - название города, ограничений на вводимые символы нет
- `latitude` (float, обязательный) — широта, от -90.0 до 90.0
- `longitude` (float, обязательный) — долгота, от -180.0 до 180.0
- `id_user` (int, обязательный) - id пользователя, для добавления в список отслеживания городов, за которыми необходимо отслеживать прогноз погоды

Логика работы:
    1. Роутер `insert_city_in_db` принимает название города, координаты, id пользователя
    2. Вызывается сервис `user_exists` из `services/work_with_db` для проверки существования пользователя в БД
    3. Вызывается сервис `get_current_weather` из `services/weather_api` для получения данных о погоде
    4. Вызывается сервис `add_weather_city` из `services/work_with_db` для добавления города для отслеживания и данных для будущего обновления погоды каждые 15 минут
    5. Вызывается сервис `add_city_user` из `services/work_with_db` для добавления в список отслеживаемых городов каждого пользователя

Реализация обновления погоды каждые 15 минут:
    Реализовано обновление при помощи асинхронной функции `update_weather_data` в `script.py`, при помощи декоратора `@asynccontextmanager` из библиотеки `contextlib` и библиотеки `asyncio`
    Логика работы:
        1. Вызывается сервис `get_all_cities` из `services/work_with_db` для получения всех городов и их координат из БД
        2. Проходимся по всем полученным значениям из БД
        3. Вызываем сервис `get_current_weather` из `services/weather_api` получения данных о погоде на момент запроса
        4. Вызывается сервис `update_weather` из `services/work_with_db` для обновления данных о погоде для каждого города
        5. Функция вызывается каждые 15 минут (прописано в последней строчке функции `update_weather_data`)
        6. Для управления фоновыми задачами при запуске и остановке сервера используется механизм `lifespan`

3. GET /cities/city - выводит список городов, за которыми пользователь отслеживает погоду

Параметры запроса:
- `id_user` (int, обязательный) - id пользователя, для которого надо вывести список городов

Логика работы:
    1. Роутер `get_list_cities` принимает параметр id_user
    2. При помощи сервиса `user_exists` из `services/work_with_db` проверяется существует ли пользователь
    3. Если не существует, возвращает ошибку, если существует, то продолжает работу
    4. Вызывается сервис `get_user_cities` из `services/work_with_db`, он возвращает все города, за которыми пользователь отслеживает прогноз погоды
    5. Если за пользователем нет закрепленных городов, возвращает об этом информацию, если есть, возвращает список городов

4. GET /weather/time - метод возвращает прогноз погоды для города по заданному времени пользователем

Параметры запроса:
- `city` (str, обязательный) - название города, ограничений на вводимые символы нет
- `hour` (int, обязательный) - время, от 0 до 23
- `temperature` (bool, необязательный) - температура
- `humidity` (bool необязательный) - влажность
- `wind_speed` (bool, необязательный) - скорость ветра
- `precipitation` (bool, необязательный) - осадки
- `id_user` (int, обязательный) - id пользователя, для добавления в список 

Логика работы:
    1. Роутер `get_weather_by_time` принимает название города, время, запращиваемые параметры погоды, id пользователя
    2. При помощи сервиса `user_exists` из `services/work_with_db` проверяется существует ли пользователь
    3. Если не существует, возвращает ошибку, если существует, то продолжает работу
    4. Вызывается сервис `get_city_coordinates` из `services/work_with_db` для получения координат города из БД
    5. Если город не найден в списке отслеживаемых городов, то возвращается информация об этом и ошибка, если найден то идем дальше по коду
    6. Далее собираются параметры, выбранные пользователем
    7. Вызывается сервис `get_hourly_weather` из `services/weather_api` для получения погоды по времени
    8. В сервисе ответ форматируется и отправляется на роутер
    9. Роутер отправляет прогноз погоды, либо ошибку

5. POST - users/id_user - метод для регистрации пользователя

Параметры запроса:
- `login` (str, обязательный) - логин пользователя

Логика работы:
    1. Вызывается сервис `get_user_id` из `services/work_with_db`
    2. Вызывается сервисс `user_exists` из `services/work_with_db`
    3. Если пользователь существует возвращает `id_user`, если нет продолжает работу
    4. Вызывается сервис `add_user` для добавления пользователя в БД
    5. Вызывается сервис `get_user_id` из `services/work_with_db`
    6. Возвращает `id_user`, либо ошибку
